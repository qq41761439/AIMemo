# AIMemo 产品与技术规划

记录时间：2026-05-12

## 1. 产品定位

AIMemo 是一个面向个人工作复盘的待办 + 周期总结工具。核心价值不是替代大型项目管理系统，而是让用户用很低成本记录每日任务，并在日、周、月、年等周期中生成结构化总结。

长期形态是多端可用：

- 桌面端：高频录入、编辑、生成总结和整理历史。
- 微信小程序：轻量查看、补记任务、移动端复盘。
- Flutter iOS/Android App：后续承接更完整的移动端体验。
- 后端服务：统一账号、云端任务、云端总结历史、免费额度和官方托管模型代理。

## 2. 当前状态

桌面端已经完成任务、标签、周期总结、模板、总结历史和本地 SQLite 持久化。自定义模型模式由桌面客户端直接调用用户配置的 OpenAI-compatible 服务。

后端已经具备基础账号与云端能力：

- 邮箱验证码登录。
- 微信小程序登录接口。
- 云端任务 CRUD，删除为软删除。
- 云端标签列表。
- 云端总结生成与历史。
- 每月免费额度。
- 官方托管模型代理。
- 本地开发可用 Postgres 持久化查看用户记录。

当前主要缺口是：桌面端本地 SQLite 数据还没有和后端云端数据形成完整同步闭环；微信账号和邮箱账号还需要完善绑定与合并策略；小程序端还没有完整任务 + 总结体验。

桌面端官方托管模式已支持 access token 自动 refresh、免费额度展示和云端总结历史查看。

## 3. 产品目标

近期目标：

- 用户可以在电脑端和微信小程序看到同一批任务和总结。
- 电脑端已有本地数据的用户，登录后可以安全合并到云端。
- 用户不需要理解本地库、云端库或同步状态，日常体验应该像一套数据。
- 官方托管模型模式能展示登录状态、免费额度和云端总结历史。

中期目标：

- 支持微信小程序完整任务闭环：新增、编辑、完成、删除、按标签筛选、查看总结历史。
- 支持桌面端离线编辑，联网后自动同步。
- 支持账号绑定：邮箱登录的桌面用户和微信登录的小程序用户可以合并到同一个用户。

长期目标：

- 形成桌面、小程序、移动 App 多端一致体验。
- 支持更完整的复盘资产管理，例如按标签、时间、项目维度回看任务和总结。
- 在保证隐私的前提下提供服务端模型能力、额度管理和付费扩展。

## 4. 账号体系

后端 `User` 是唯一用户主体。一个用户可以绑定多个登录方式：

- 邮箱：桌面端优先使用。
- 微信 OpenID：微信小程序使用。
- 微信 UnionID：如果后续开放多个微信应用，应优先用 UnionID 合并身份。

原则：

- 不要让同一个真实用户因为邮箱登录和微信登录产生两套长期数据。
- 账号绑定前可以允许各自独立存在，但绑定时必须提供明确的合并流程。
- 合并账号时，任务、总结、额度和会话都应归并到最终保留的 `User.id`。
- 绑定敏感操作应要求当前登录态有效，必要时二次验证邮箱验证码。

建议流程：

1. 桌面端用户用邮箱登录。
2. 用户在桌面端或小程序端发起绑定微信。
3. 后端确认微信身份。
4. 如果微信身份已对应另一个用户，进入账号合并流程。
5. 合并完成后，两端都指向同一个 `User.id`。

## 5. 数据同步方案

推荐架构：Postgres 后端作为最终数据源，桌面 SQLite 和小程序本地缓存作为客户端缓存。

### 5.1 数据源职责

后端 Postgres：

- 保存跨端共享数据。
- 负责账号、权限、额度和服务端模型代理。
- 作为同步冲突的最终裁决方。

桌面 SQLite：

- 保存本机任务、模板、总结历史和 UI 设置。
- 支持离线新增、编辑、完成和删除任务。
- 保存同步状态和云端 ID 映射。

微信小程序本地缓存：

- 缓存最近任务、标签和总结历史。
- 初期可以只做在线读写，后续再支持离线编辑。

### 5.2 任务同步字段

桌面端本地任务需要补充同步相关字段：

```text
localId       本地 SQLite 自增 ID
cloudId       后端 Task ID
clientId      客户端生成的 UUID，用于防止重复上传
syncStatus    synced / pendingCreate / pendingUpdate / pendingDelete / conflict
createdAt     开始时间
updatedAt     最后更新时间
completedAt   完成时间
deletedAt     软删除时间
```

后端任务建议保留：

```text
id
userId
body
tags
isCompleted
createdAt
completedAt
updatedAt
deletedAt
clientId
```

`clientId` 用于幂等创建：客户端重复提交同一条本地任务时，后端应返回同一条云端记录，而不是创建重复任务。

### 5.3 同步流程

每个客户端保存 `lastSyncedAt`。

一次同步建议按这个顺序执行：

1. 上传本地 pending 改动。
2. 后端按 `clientId` 和 `cloudId` 幂等处理新增、更新和删除。
3. 拉取服务端 `updatedAt > lastSyncedAt` 的改动。
4. 合并到本地缓存。
5. 更新 `lastSyncedAt`。

可以先复用现有任务 API 与 `updatedAfter` 参数，后续再收敛为统一接口：

```text
POST /sync
GET /sync?since=...
```

### 5.4 删除策略

所有端都使用软删除：

```text
deletedAt
```

原因是同步需要知道“这条记录被删除过”。如果物理删除，其他端可能会把旧数据重新上传回来。

### 5.5 冲突策略

第一版采用 Last Write Wins：

- `updatedAt` 更新的一方覆盖旧数据。
- 删除也视为一次更新。
- 如果一端删除后，另一端基于旧版本提交更新，默认删除优先。
- 如果另一端是在删除之后又编辑，应按更新时间恢复为非删除状态，但需要在产品上谨慎处理。

暂不建议第一版引入 CRDT。任务数据结构简单，复杂冲突 UI 会拖慢上线。

## 6. 标签策略

标签继续从任务中派生，不单独作为跨端主数据。

原则：

- 标签存放在 `Task.tags` 中。
- 只展示仍有关联非删除任务的标签。
- 标签排序按最近关联任务靠前。
- 无任务关联的标签不显示。

这样可以避免空标签、脏标签和跨端标签表同步复杂度。

## 7. 总结同步

总结记录应进入云端同步，因为用户会自然期待在电脑端生成的总结能在小程序看到。

后端保存字段：

```text
id
userId
periodType
periodLabel
periodStart
periodEnd
tags
output
model
createdAt
```

隐私原则：

- 云端历史保存总结输出、周期、标签、模型等元数据。
- 不保存完整 prompt 快照。
- 自定义模型 API Key 不进入后端、不进入 SQLite、不进入 git。

## 8. 模板与设置

建议同步：

- 总结模板。
- 默认总结周期。
- 与用户复盘体验直接相关的偏好。

建议不跨端同步：

- 自定义模型 API Key。
- 本机窗口宽度。
- 桌面端安全存储中的 access token / refresh token 原文。
- 与具体设备有关的 UI 状态。

## 9. 桌面端首次登录迁移

桌面端已有 SQLite 本地数据时，首次登录云端账号后需要明确处理。

推荐默认方案：合并本机和云端数据。

可选项：

- 合并本机和云端数据。
- 将本机数据上传到云端。
- 从云端覆盖本机。

合并时要注意：

- 本地任务上传前生成 `clientId`。
- 上传成功后保存 `cloudId`。
- 使用 `clientId` 或内容哈希降低重复导入概率。
- 总结历史也应合并，但不上传完整 prompt。

## 10. API 规划

已有接口可以支撑早期多端闭环：

- `POST /auth/email/start`
- `POST /auth/email/verify`
- `POST /auth/wechat/mini-program/login`
- `POST /auth/refresh`
- `GET /me`
- `GET /me/quota`
- `GET /tasks`
- `POST /tasks`
- `PATCH /tasks/:id`
- `DELETE /tasks/:id`
- `GET /tags`
- `POST /summaries/generate`
- `GET /summaries`
- `GET /client/config`

建议新增或增强：

- 账号绑定接口：绑定邮箱、绑定微信、查询绑定状态。
- 账号合并接口：处理同一个真实用户的邮箱账号和微信账号合并。
- 同步接口：批量上传本地改动，批量拉取服务端变更。
- 任务幂等创建：支持 `clientId`。
- 总结历史增量拉取：支持 `createdAfter` 或统一 sync cursor。

## 11. 分阶段路线图

### 阶段一：后端持久化与可观察性

- 本地 Postgres 开发环境。
- Prisma Studio 查看用户和云端数据。
- 命令行查看用户列表。
- 后端测试覆盖账号、任务、总结、额度。

状态：已基本完成。

### 阶段二：桌面端官方托管模式完善

- access token 自动 refresh。
- 免费额度展示。
- 云端总结历史视图。
- 托管模型错误状态更清晰。

状态：已基本完成。

### 阶段三：桌面端本地数据云同步

- SQLite 增加同步字段。
- 登录后支持本地数据合并到云端。
- 后端支持 `clientId` 幂等创建。
- 桌面端支持增量拉取云端任务与总结。
- 基础冲突策略采用 Last Write Wins。

### 阶段四：微信小程序任务闭环

- 微信登录接入真实小程序配置。
- 小程序读取云端任务、标签和总结历史。
- 小程序新增、编辑、完成、删除任务。
- 小程序生成或查看周期总结。

### 阶段五：账号绑定与合并

- 桌面邮箱账号绑定微信。
- 小程序微信账号绑定邮箱。
- 已存在双账号时提供合并流程。
- 合并后统一 `User.id` 和云端数据。

### 阶段六：移动 App 与商业化基础

- Flutter iOS/Android App 复用云端 API。
- 免费额度、付费额度和套餐策略。
- 服务端模型调用审计、限流和成本监控。

## 12. 风险与原则

主要风险：

- 账号合并错误导致数据归属混乱。
- 离线同步重复创建任务。
- 删除同步处理不当导致任务被恢复。
- API Key 或 token 被误写入 SQLite、README、测试或 git。
- 小程序和桌面端同时上线同步逻辑，调试面过大。

执行原则：

- 先后端持久化，再多端同步。
- 先在线闭环，再离线编辑。
- 先 Last Write Wins，再考虑复杂冲突处理。
- 先合并本地数据，再做覆盖类危险操作。
- 所有真实密钥只放系统安全存储或服务端环境。
